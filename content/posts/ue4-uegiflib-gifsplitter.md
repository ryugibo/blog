# 왜 이런짓을 했는가

예전에 `UEgiflib`라는 이름으로 gif 파일을 언리얼 엔진 에디터에 가져오기 하면 텍스쳐 > 스프라이트 > 플립북까지 생성하는 플러그인을 만들었다.

마켓플레이스에 gif 파일 자체를 하나의 텍스쳐로 사용하는 플러그인 Animated Texture with GIF Importer(https://www.unrealengine.com/marketplace/ko/product/animated-texture-with-gif-importer) 이 무료로 나와서 이제 약간 개발 의욕이 떨어지기도 했었고, 실제 프로젝트에 사용해보지 않다보니 방치되고 있었다.

이번에 2D 프로젝트를 잠깐 해보다가 플러그인을 살려봤는데 실제 사용해보니 애매한 부분이 있어서 구조를 바꿔보고, 바꾸는 김에 라이브러리도 바꿔보고 하다가 그냥 새로 플러그인을 만들었다.

이름은 `GifSplitter` .. 언리얼 엔진 플러그인인데 매번 `UE`를 붙이는게 뭔가 이상해서 `UE`는 떼버렸고, Paper2D관련 로직을 드러내면서 결국 단순히 gif 파일을 구성하는 각 프레임을 개별 텍스쳐로 임포트하는 플러그인이 되서, 이렇게 지었다.

대충 변경한 이유를 적어본다.

# 라이브러리를 변경 (giflib 👉 gif_load)

기존에 giflib라는 gif 라이브러리를 대충 검색하면 바로 나오는 라이브러리를 사용했는데 구지 기존 라이브러리를 사용할 필요가 없어 보여서 좀 찾아보다가 헤더 파일 하나에 gif 파일을 읽을 수 있는 라이브러리가 있어서 사용했다.

자세한 이유들은 아래에 설명한다.

## giflib의 모든 기능이 필요하지 않다.

giflib는 gif 파일을 읽을 뿐아니라 생성하는 부분까지 gif 파일을 다루는 모든 내용이 들어가있는데, 이 플러그인은 gif 파일을 쪼개서 개별 텍스쳐로 언리얼 엔진에 넣는게 목적이기 때문에 gif 파일을 읽는 부분까지만 기능이 있으면 됐다.

라이브러리 크기가 크지는 않았지만..

## 코드 관리 어려움

이 부분인 기존에 작업을 잘못 기준을 잡고 작업을 해놨던것 같다. giflib 수정한 걸 별도 저장소에 두고 서브 모듈로 넣어서 작업했으면 giflib 변경점을 제대로 관리할 수 있었을텐데, 그냥 복사 붙여넣기로 코드들을 다른 플러그인 코드와 함께 둬서 구분이 미묘해져있었다.

심지어 중간에 라이브러리 구조를 변경한다고 한번 위치를 크게 변경하면서 커밋을 뭉쳐서 머지해놔서 라이브러리 코드를 수정한 걸 제대로 파악하기 어려운 상태였다.

아무튼, 라이브러리를 미리 컴파일 된 상태로 플러그인에 넣거나 라이브러리 코드를 `git submodule`로 포함시키고자 했다.

## gif_load로..

라이브러리 이름도 짧고 단일 헤더파일로 구성된 라이브러리라 맘에 들어서 냅다 때려박고 컴파일 되자마자 그대로 적용했다.

C++을 지원하는 C 코드 기반이라 좀 찝찝한 부분이 있지만 혹시 모르니 클론을 떠두고 워닝만 조금 수정하고 사용했다.

# Paper2D 관련 로직 제거.

이 부분은 실제 gif 파일을 임포트 해보면서 불편한 부분이 있어서 관련 부분을 제거했다.

Animated Texture with GIF Importer 플러그인을 2D 프로젝트에 사용하지 않은 이유와 비슷한데 Paper2D 나 PaperZD 기능을 제대로 사용할 수 없어서 그 플러그인 사용은 하지 않게 되었다.

자세한 이유는 이어서 설명

## Paper2D 에셋에 저장되는 데이터는 gif 파일에 포함되지 않는 데이터

처음에 gif를 플립북까지 변환해서 처리하면서 gif 파일에 프레임당 재생 속도까지 반영해서 플립북에 반영하도록 작업을 했었는데, 실제 게임용 gif 파일의 경우 atlas화 가능한 이미지 파일이다보니 각 프레임이 어느정도 재생할지는 gif 파일에서 관리하기보다는 게임 개발하면서 조절하는 경우가 더 편하다는 것을 깨달았다.

그리고 2D 스프라이트에 소켓을 박을 경우 다시 gif를 넣으면서 스프라이트와 플립북이 새로 세팅되면 소켓 정보가 날아가는 경우까지 발생할 수 있었다. 이 경우 리임포트 부분을 작업하면 될텐데 파일 이름이 유지되어야 하는 등 작업 과정에 제약을 두게 되서 좋지 않다고 판단해서 과감히 Paper2D관련 처리는 다 제거했다.

# bmp 형태로 처리하던 부분을 tga로 변경

변경 적용한 라이브러리의 `README.md`를 살펴보면 tga로 변환하는 걸로 추정되는 샘플 코드가 작성되어있어서 도움이 됐다.

기존에 `UTextureFactory`에 정의된 함수를 사용하지 않고 직접 `UTextureFactory`와 같은 로직을 중복해서 선언해서 처리한 로직이 있었는데, tga 파일 구조로 버퍼를 구성해서 `UTextureFactory` 함수를 사용해보니 너무나 잘 동작해서 놀랬다. 그대로 적용했다.

처음엔 이런 생각은 하지도 못했지만..

# 결론

https://github.com/ryugibo/GifSplitter 에 플러그인은 공개되어있고, C++ 컴파일이 가능한 환경이 필요하다.

작업 전 플러그인은 https://github.com/ryugibo/UEgiflib 이쪽, `UEgiflib`는 예전에 macOS에서 컴파일을 테스트했는데 이번건 테스트하지 않았다.
